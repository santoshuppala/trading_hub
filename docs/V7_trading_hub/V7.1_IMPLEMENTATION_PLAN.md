## ITEM 1: Persistent ORDER_REQ Dedup Set (P2)

### Problem
`SmartRouter._routed_event_ids` is in-memory only. On Core restart, the entire dedup set is lost. If Redpanda replays an ORDER_REQ that was already routed (but not committed due to crash), the triple safety net relies entirely on Layer 3 (`client_order_id` at Alpaca). Tradier has no equivalent — a replayed Tradier ORDER_REQ creates a duplicate real order.

### Current Code
```python
# monitor/smart_router.py line 99
self._routed_event_ids: dict = {}  # in-memory, lost on restart
```

### Fix
Persist `_routed_event_ids` to disk using SafeStateFile. Load on startup. Write on every new routing (debounced — skip if hash unchanged).

```python
# monitor/smart_router.py
self._dedup_sf = SafeStateFile('data/routed_event_ids.json', max_age_seconds=3600)

def _on_order_req(self, event):
    ...
    # After adding to _routed_event_ids:
    if len(self._routed_event_ids) % 10 == 0:  # debounce: save every 10 routes
        self._dedup_sf.write({'ids': list(self._routed_event_ids.keys())[-5000:]})

# On startup:
data, _ = self._dedup_sf.read()
if data and self._dedup_sf.is_today(data):
    self._routed_event_ids = {eid: 0 for eid in data.get('ids', [])}
```

### Files
- `monitor/smart_router.py` — add SafeStateFile for dedup persistence

### Tests
- Write dedup set → restart (new SmartRouter instance) → verify replayed ORDER_REQ is skipped
- Verify debounced save (not every single route)
- Verify stale dedup (yesterday's) discarded on startup

## ITEM 3: BaseBroker.has_position() Interface (P4)

### Problem
`SmartRouter._select_broker()` checks `hasattr(broker, 'has_position')` — fragile. New broker implementations may forget to implement it.

### Current Code
```python
# monitor/smart_router.py line 224
if hasattr(broker, 'has_position'):
    if broker.has_position(ticker):
        ...
elif hasattr(broker, '_client') and broker._client:  # legacy Alpaca fallback
```

### Fix
Add `has_position(ticker) -> bool` as an abstract method on `BaseBroker`. Implement in `AlpacaBroker` (delegates to `_client.get_open_position`) and `TradierBroker` (delegates to position API). `PaperBroker` returns False (no real positions).

```python
# monitor/brokers.py
class BaseBroker(ABC):
    @abstractmethod
    def has_position(self, ticker: str) -> bool:
        """Check if this broker holds an open position for ticker."""
        ...

class AlpacaBroker(BaseBroker):
    def has_position(self, ticker: str) -> bool:
        try:
            pos = self._client.get_open_position(ticker)
            return pos and int(float(pos.qty or 0)) > 0
        except Exception:
            return False
```

### Files
- `monitor/brokers.py` — add abstract method + AlpacaBroker/PaperBroker implementations
- `monitor/tradier_broker.py` — add TradierBroker implementation
- `monitor/smart_router.py` — remove `hasattr` fallback, call directly

### Tests
- AlpacaBroker.has_position() with mock client
- TradierBroker.has_position() with mock API
- PaperBroker.has_position() returns False
- SmartRouter SELL routing uses has_position() (no hasattr check)

## ITEM 4: Generalized Engine Config (P4)

### Problem
Adding a new engine requires manual entries in `config.py` for `{ENGINE}_MAX_POSITIONS`, `{ENGINE}_TRADE_BUDGET`, `{ENGINE}_ORDER_COOLDOWN`, `{ENGINE}_MAX_DAILY_LOSS`. No template or auto-discovery for config.

### Current Code
```python
# config.py lines 98-109 — each engine hardcoded separately
POP_MAX_POSITIONS = int(os.getenv('POP_MAX_POSITIONS', 15))
PRO_MAX_POSITIONS = int(os.getenv('PRO_MAX_POSITIONS', 15))
# ... 20+ lines of per-engine config
```

### Fix
Add `EngineConfig` dataclass and `get_engine_config(name)` factory function. Engine-specific env vars auto-discovered by prefix convention. Defaults provided. Existing hardcoded values preserved as backward compat.

```python
# config.py
@dataclass
class EngineConfig:
    max_positions: int = 5
    trade_budget: int = 1000
    order_cooldown: int = 300
    max_daily_loss: float = -2000.0
    paper_trading: bool = True

_ENGINE_DEFAULTS = {
    'core': EngineConfig(max_positions=5, trade_budget=1000, max_daily_loss=-10000),
    'pro':  EngineConfig(max_positions=15, trade_budget=1000),
    'pop':  EngineConfig(max_positions=15, trade_budget=2000),
    'options': EngineConfig(max_positions=5, trade_budget=2000, max_daily_loss=-3000),
}

def get_engine_config(name: str) -> EngineConfig:
    """Get config for engine, reading {NAME}_MAX_POSITIONS etc from env."""
    defaults = _ENGINE_DEFAULTS.get(name, EngineConfig())
    prefix = name.upper()
    return EngineConfig(
        max_positions=int(os.getenv(f'{prefix}_MAX_POSITIONS', defaults.max_positions)),
        trade_budget=int(os.getenv(f'{prefix}_TRADE_BUDGET', defaults.trade_budget)),
        ...
    )
```

### Files
- `config.py` — add EngineConfig dataclass + get_engine_config() factory
- Existing hardcoded variables preserved (backward compat)

### Tests
- get_engine_config('pro') returns correct defaults
- get_engine_config('arbitrage') returns EngineConfig defaults (new engine, no env vars)
- Env var override: set ARBITRAGE_MAX_POSITIONS=10 → reflected in config

## ITEM 5: Memory Monitoring (P3)

### Problem
No RSS/heap/GC metrics. `MetricsWriter` only captures EventBus queue-based metrics. Long-running sessions (6.5 hours daily) could leak memory with no detection.

### Fix
Add `resource.getrusage()` and `sys.getsizeof` metrics to `MetricsWriter._write()`.

```python
# monitor/metrics.py
import resource, gc

def _write(self):
    m = self._bus.metrics()
    rusage = resource.getrusage(resource.RUSAGE_SELF)
    data = {
        ...existing metrics...,
        'memory': {
            'rss_mb': rusage.ru_maxrss / 1024 / 1024,  # macOS: bytes, Linux: KB
            'gc_objects': len(gc.get_objects()),
            'gc_collections': gc.get_stats(),
        },
    }
```

Alert if RSS exceeds threshold (e.g., 2GB):
```python
if rusage.ru_maxrss > _RSS_ALERT_THRESHOLD:
    send_alert(None, f"MEMORY: RSS={rss_mb:.0f}MB exceeds {threshold}MB", severity='WARNING')
```

### Files
- `monitor/metrics.py` — add memory metrics to `_write()`

### Tests
- Metrics file contains `memory.rss_mb` field
- RSS value is > 0 (sanity check)

## ITEM 6: IPC Consumer Backpressure (P3)

### Problem
IPC consumer polls Redpanda continuously regardless of Core's EventBus queue depth. If Core is slow (DB circuit breaker open, broker hanging), consumer keeps pulling messages into `_ipc_inbox`, which eventually fills to `maxsize=500` and drops messages.

### Fix
Check EventBus `system_pressure` before draining inbox. Pause if > 80%.

```python
# scripts/run_core.py
def _drain_ipc_inbox():
    # V7.1: Check EventBus pressure before draining
    pressure = monitor._bus.metrics().system_pressure
    if pressure > 0.80:
        log.warning("[IPC] EventBus pressure %.0f%% — pausing inbox drain", pressure * 100)
        return  # don't drain this tick — let bus catch up
    ...existing drain logic...
```

### Files
- `scripts/run_core.py` — add pressure check in `_drain_ipc_inbox()`

### Tests
- Mock bus with high pressure → drain skipped
- Normal pressure → drain proceeds
- Pressure drops below threshold → drain resumes

## ITEM 9: Cross-EventType Ordering Invariant (P5)

### Problem
EventBus explicitly does NOT guarantee cross-EventType ordering (ORDER_REQ and FILL use separate worker pools). This is by design but undocumented as an architectural invariant. Developers may write handlers that assume ORDER_REQ always processes before FILL.

### Fix
Add explicit documentation in EventBus and architecture docs:

```python
# monitor/event_bus.py — docstring addition
"""
ARCHITECTURAL INVARIANT: Cross-EventType ordering is NOT guaranteed.
ORDER_REQ and FILL use separate worker pools. A fast broker FILL can
be dequeued before ORDER_REQ processing completes. Design handlers
to be order-independent across EventTypes.

Within a single (EventType, ticker): events are FIFO-ordered.
"""
```

### Files
- `monitor/event_bus.py` — add docstring to class header
- `docs/V7_trading_hub/V7_ARCHITECTURE_AND_DESIGN.md` — add invariant section

## ITEM 10: NTP Drift Check (P5)

### Problem
All processes use `datetime.now(ET)` independently. If the system moves to multi-host deployment, clock drift causes EOD gates to fire at wrong times, cooldown timers to be wrong, and state file date checks to fail.

### Current Mitigation
Single-host deployment — all processes share the system clock. Risk is theoretical.

### Fix
Add startup NTP drift check (best-effort, non-blocking):

```python
# lifecycle/core.py — add to startup()
def _check_clock_drift(self):
    """V7.1: Warn if system clock drifts from NTP."""
    try:
        import ntplib
        c = ntplib.NTPClient()
        r = c.request('pool.ntp.org', version=3, timeout=2)
        drift_ms = abs(r.offset) * 1000
        if drift_ms > 1000:  # > 1 second drift
            log.warning("CLOCK DRIFT: %.0fms from NTP. Trading timers may be inaccurate.", drift_ms)
    except Exception:
        pass  # NTP unavailable — single-host assumed
```

### Files
- `lifecycle/core.py` — add `_check_clock_drift()` to startup

### Tests
- Mock NTP response with 0ms drift → no warning
- Mock NTP response with 5000ms drift → warning logged
- NTP unavailable → silent pass (no crash)

## ITEM 11: DataSourceCollector in Supervised Mode (P2)

### Problem
`DataSourceCollector` (Fear & Greed, FRED, Finviz, SEC EDGAR, Polygon, Yahoo) runs ONLY in `run_monitor.py` (monolith mode). In supervised mode (`supervisor.py` → `run_core.py`), these 6 data sources are NOT collected. The V6 docs incorrectly labelled them as "Active" — they're only active in monolith mode, which is no longer used.

**Active in supervised mode:** Tradier (Core), Benzinga (Pop), StockTwits (Pop)
**NOT active in supervised mode:** Fear & Greed, FRED, Finviz, SEC EDGAR, Polygon, Yahoo

### Fix
Create `scripts/run_data_collector.py` — a new satellite process managed by supervisor. Runs `DataSourceCollector` from `data_sources/collector.py` with the same collection schedule (session start, periodic 10 min, session end).

Supervisor auto-discovers it via `run_*.py` pattern (V7 P4-2). No supervisor code changes needed.

```python
# scripts/run_data_collector.py
from data_sources.collector import DataSourceCollector

collector = DataSourceCollector()
collector.collect_session_start()
while market_open:
    collector.collect_periodic()
    time.sleep(600)  # 10 min
collector.collect_session_end()
```

### Files
- `scripts/run_data_collector.py` — **NEW** satellite process
- No changes to supervisor (auto-discovered)
- No changes to existing data source files

### Tests
- Verify `run_data_collector.py` exists → supervisor discovers it
- Fear & Greed returns data (API call)
- Finviz screener returns data
- SEC EDGAR returns data
